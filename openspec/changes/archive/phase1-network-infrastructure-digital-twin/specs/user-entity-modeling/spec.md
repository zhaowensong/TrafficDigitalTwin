## ADDED Requirements

### 需求:用户轨迹数据接入
系统必须接入研究团队提供的真实用户轨迹数据（user_data_shanghai_v1.zip）。

#### 场景:解压并加载轨迹数据
- **当** 应用启动时
- **那么** 必须从 data/user_data/ 目录加载 trajectories.json
- **并且** 数据包含1万用户、7天连续轨迹、30分钟粒度
- **并且** 每条轨迹记录包含：经纬度、连接基站、位置类型、APP使用、流量消耗、会话时长、信号强度、到基站距离

#### 场景:轨迹数据结构
- **当** 读取单条轨迹记录
- **那么** 必须包含以下字段：
  - user_id: 字符串，用户唯一标识
  - timestamp: 时间戳，30分钟粒度
  - latitude/longitude: 数值，真实经纬度
  - connected_bs: 字符串，连接的基站ID
  - location_type: 字符串，位置类型（home/work/leisure/transit）
  - app_usage: 对象，当前APP使用情况
  - traffic_volume: 数值，流量消耗（MB）
  - session_duration: 数值，会话时长（秒）
  - signal_strength: 数值，信号强度（dBm）
  - distance_to_bs: 数值，到服务基站距离（米）

### 需求:用户画像数据
系统必须接入用户结构化画像和文本画像。

#### 场景:结构化画像
- **当** 加载用户画像数据
- **那么** 必须从 user_profiles_en.json 加载
- **并且** 每个用户画像包含：角色、年龄段、活跃时段、偏好类别、锚点基站等

#### 场景:文本画像
- **当** 查询单个用户画像
- **那么** 必须支持从 profiles_txt/ 目录加载该用户的英文文本画像
- **并且** 1万份用户画像文本不重复

### 需求:用户数据加载性能
系统必须支持高效加载大规模用户数据。

#### 场景:加载性能要求
- **当** 应用启动时
- **那么** trajectories.json（~100MB）加载应在合理时间内完成
- **并且** 支持按需加载（按用户ID/按区域/按时间段筛选）
- **并且** 不应阻塞服务器其他API的响应

#### 场景:按区域查询用户
- **当** 前端请求特定区域的用户分布
- **那么** 必须支持按经纬度范围筛选用户
- **并且** 响应时间应在2秒内

### 需求:用户-基站关联
用户轨迹数据必须与现有基站数据建立关联。

#### 场景:基站连接映射
- **当** 加载用户轨迹数据
- **那么** 每个轨迹点的 connected_bs 必须对应 base2info_extended.json 中的真实基站
- **并且** 支持查询某基站在某时段连接的所有用户
