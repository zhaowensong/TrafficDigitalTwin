## 上下文

当前项目是一个基于 Flask 的基站流量预测与能源优化系统，基础功能已修复：
- 服务器启动正常（10-20秒）
- 前端页面正常加载，已改造为 Electron 桌面应用
- API 响应优化（~200KB）
- C/S 架构已完成（Electron前端 + Flask后端）

现有基站数据（base2info_extended.json）包含：id, loc, 天线参数, 容量模型, 状态监控, 空间特征。

研究团队已提供真实用户数据（user_data_shanghai_v1.zip）：
- 1万用户×7天×30分钟粒度 = 336万轨迹点
- 含真实经纬度、基站连接、APP使用、流量消耗、信号强度
- 含结构化画像 + 英文文本画像

## 决策

### 1. 基站属性扩展方式
**决策**: 扩展 base2info.json，新增字段用假数据填充
**理由**: 
- 保持现有数据结构兼容
- 假数据格式与真实数据一致，后续可替换
- 不破坏现有 API 响应

### 2. 用户数据接入方式
**决策**: 直接接入研究团队提供的真实用户数据（user_data_shanghai_v1.zip）
**理由**:
- 数据已包含完整用户属性：轨迹、画像、APP使用、流量消耗
- 基于真实上海基站坐标生成，与现有基站数据匹配
- 无需再写 user_generator.py 生成假数据
- 数据结构：trajectories.json（轨迹）+ user_profiles_en.json（画像）+ profiles_txt/（文本画像）

### 3. APP模型定义
**决策**: 基于真实用户数据中的 APP 使用记录，Python 字典/类定义流量特征
**理由**:
- 用户数据中已包含 APP 使用情况、流量消耗、会话时长
- 可从真实数据提取统计特征，辅以平台需求中的分类标准
- 四类分类：视频、社交、游戏、浏览

### 4. 前端架构: Web vs 桌面应用
**决策**: 使用 Electron 将前端打包为桌面应用
**理由**:
- 复用现有 HTML/JS/CSS 代码，最小改动
- 支持 C/S 架构：后端部署在服务器，前端在普通电脑
- 用户体验更好：双击运行，无需浏览器
- 支持远程服务器配置，灵活部署
**替代方案**: PyQt 重写前端 - 工作量大，不划算

### 5. C/S架构通信
**决策**: HTTP REST API 通信，桌面客户端直连后端
**理由**:
- 现有 API 无需修改
- 跨网络访问简单
- 配置灵活，支持本地/远程后端
**替代方案**: WebSocket 实时通信 - 当前不需要实时推送
